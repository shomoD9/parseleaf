# Architecture

## Opening

Parseleaf is a project about taking text and making it AI-usable. In its current form, that philosophy is embodied as a local command-line translator from EPUB into a semantically structured markdown workspace. It does not treat a book as a single extractable text stream. Instead, it treats the EPUB as a publication package whose own internal claims about structure matter: tables of contents, prefaces, chapters, appendices, notes, and other named regions are all different kinds of textual territory, and the point of the tool is to preserve those territories as separate markdown files rather than flattening them into one large output.

The resulting shape is meant for downstream reasoning. A human can still read the produced markdown, but the system is optimized for cases where a book will be searched, chunked, indexed, embedded, or otherwise processed by tools that benefit from semantically coherent files. That is why Parseleaf CLI keeps a manifest, carries section metadata into YAML frontmatter, and extracts assets into a stable companion directory. The output is designed to remain legible to people while also becoming easier for software to work with. As the project began preparing for wider distribution, that same concern for usable shape extended to delivery itself: the codebase now treats packaging as part of the product, not as an afterthought, which is why there is a curated npm package, a standalone binary build path, and an automated release pipeline aimed at npm, GitHub Releases, and Homebrew.

## Ontology

The central entity here is the parsed publication. A parsed publication is still the original EPUB, but lifted into a working form that exposes its metadata, manifest, spine, navigation items, semantic landmarks, guide references, and spine documents. That object is the system’s internal version of the book as a package rather than a blob. The next major entity is the section plan. A section plan is the converter’s interpretation of the book’s top-level meaning: which parts belong together, where a chapter begins, whether a document is a preface or an appendix, and which source files together should become one markdown output.

Within those sections, the most practical unit is the part. A part is a slice of one spine document that belongs to a section. This matters because an EPUB chapter can span multiple XHTML files, and in more complicated cases several semantic sections can even begin inside one source file. Finally there are assets, which are images or related non-HTML resources that the markdown still needs. Assets are not the point of the output, but they travel with the text so links and illustrations do not collapse when the book leaves the EPUB container.

## Geography

The repository root holds the public and operational framing. `README.md` explains Parseleaf from the outside, makes clear that Parseleaf CLI is the current shipped interface, and now documents both the npm install path and the Homebrew install path. `package.json` declares the Node 22 package as `parseleaf`, exposes the `parseleaf` CLI binary, records the repository metadata that point back to the canonical GitHub project over SSH and HTTPS, and now also defines the curation rules that decide what actually gets published to npm. `package-lock.json` freezes that dependency graph so the same toolchain can be reproduced, including the extra release-time tooling used for binary builds. `LICENSE` gives the repository an explicit MIT legal boundary so package registries and downstream users are not left inferring intent. `tsconfig.json` defines the TypeScript compilation boundary, while `vitest.config.ts` tells the test runner where the test files live. `.gitignore` excludes generated directories such as `node_modules/`, `dist/`, `output/`, and the temporary release workspace under `.release/`. `ARCHITECTURE.md` and `DEVLOG.md` are the narrative map and chronological ledger that the repository instructions require.

The `.github/` directory now contains the delivery automation. Inside it, `.github/workflows/release.yml` is the release conveyor that turns a semver tag into a validated npm package, two macOS standalone binaries, a GitHub Release with attached artifacts and checksums, and an updated Homebrew tap formula. The `scripts/release/` directory holds the local logic that makes that automation readable rather than burying everything inside YAML. `scripts/release/build-sea.mjs` bundles the CLI into one CommonJS entry, prepares the SEA blob, injects it into a copied `node` executable with `postject`, re-signs the macOS binary, and wraps the result as the tarball that GitHub Releases and Homebrew will consume. `scripts/release/render-homebrew-formula.mjs` turns release metadata into the exact `parseleaf.rb` formula that the dedicated Homebrew tap needs.

The `src/` directory is the implementation heart of the system. `src/cli.ts` is the thin terminal-facing entrypoint that parses arguments and hands control to the orchestrator. `src/index.ts` is that orchestrator: it ties parsing, section inference, and output writing into one callable conversion function. `src/types.ts` names the shared data structures that let the stages of the pipeline talk to each other coherently. `src/utils.ts` contains the low-level helpers for path resolution, href splitting, whitespace cleanup, slug generation, and safe output naming. `src/parse-epub.ts` is the archive and metadata reader that opens the ZIP container, resolves the package document, and loads the manifest, spine, navigation, landmarks, guide references, and spine XHTML files. `src/sectioning.ts` interprets that parsed book into semantic sections by following top-level navigation when it exists and falling back to meaningful spine documents when it does not. `src/markdown.ts` rewrites links, preserves anchors, extracts asset references, and converts cleaned HTML fragments into markdown. `src/output.ts` writes the numbered markdown files, copies only the assets that were actually referenced, and serializes the root `manifest.json`. `src/turndown.d.ts` is a small type declaration shim that gives the TypeScript build enough information to use the markdown conversion library cleanly.

The `test/` directory is split between helpers, behavioral tests, and fixture books. `test/helpers.ts` assembles readable fixture trees into temporary EPUB archives so the tests can exercise the real ZIP-based parser without storing opaque binaries, and it now resolves those fixtures relative to itself so a project rename does not silently break the suite. `test/utils.test.ts` verifies the small path and naming primitives. `test/sectioning.test.ts` tests the semantic split logic directly, especially the multi-file chapter case. `test/convert.test.ts` runs the full end-to-end conversion path and checks manifests, markdown output, asset copying, NCX fallback behavior, and degraded-book fallback behavior. Under `test/fixtures/`, `epub3/` is a modern EPUB 3 sample whose `META-INF/container.xml`, `OEBPS/content.opf`, `OEBPS/nav.xhtml`, `OEBPS/contents.xhtml`, `OEBPS/preface.xhtml`, `OEBPS/chapter-1a.xhtml`, `OEBPS/chapter-1b.xhtml`, `OEBPS/appendix.xhtml`, and `OEBPS/images/diagram.svg` together test landmarks, semantic sections, a multi-file chapter, tables, and asset extraction. `epub2/` contains a legacy package whose `OEBPS/content.opf`, `OEBPS/toc.ncx`, `OEBPS/cover.xhtml`, `OEBPS/introduction.xhtml`, `OEBPS/chapter-one.xhtml`, and `OEBPS/notes.xhtml` prove the NCX path still works when there is no EPUB 3 nav document. `degraded/` holds a sparse book whose `OEBPS/content.opf`, `OEBPS/titlepage.xhtml`, `OEBPS/chapter-a.xhtml`, and `OEBPS/chapter-b.xhtml` force the fallback logic to infer sections directly from the spine and document headings. The generated `dist/` directory is the compiled output of the TypeScript source; it is still a build artifact rather than authored source, but it now also matters because only `dist/src/` is the published slice that goes to npm while the SEA build path bypasses `dist/` and bundles directly from source for standalone binaries.

## Flow

The primary flow begins with `src/cli.ts`. A user runs `parseleaf convert <input.epub> [--out <dir>]`, the CLI validates that the command matches the expected contract, and then it delegates almost immediately to `convertEpub` in `src/index.ts`. From there `src/parse-epub.ts` opens the EPUB as a ZIP archive, reads `META-INF/container.xml` to find the package document, parses the OPF metadata and reading order, and then tries to discover a top-level table of contents either through an EPUB 3 nav document or an EPUB 2 NCX file. At that point the system has moved from “file on disk” to “book as structured package.”

The second flow belongs to `src/sectioning.ts`. If the publication exposes usable top-level navigation, those entries become the candidate semantic boundaries. The module maps each boundary into the spine, slices the relevant XHTML documents into section parts, classifies each result as a contents page, preface, chapter, appendix, notes section, and so on, and merges adjacent spine files when the TOC says they are really one chapter. If the navigation is absent or broken, the module falls back to the meaningful documents in the spine and derives titles from headings or document titles. Either way, it produces a section plan that says what files should exist before any markdown is written.

The third flow is the translation from section plan to output workspace. `src/output.ts` assigns stable numbered filenames and builds a map from source XHTML targets to output markdown files. `src/markdown.ts` then rewrites internal links so they point at the generated markdown layout rather than the original EPUB files, inserts raw anchor tags so fragment links still have destinations, rewrites image references into the output `assets/` tree, and feeds the cleaned HTML into Turndown to become markdown. Back in `src/output.ts`, the tool writes each markdown file with YAML frontmatter, copies only the assets the markdown actually references, and writes `manifest.json` so downstream software can see the resulting order, titles, source mappings, and asset inventory in one place.

There is now a fourth flow, and it begins not with a user converting a book but with a maintainer cutting a release. A semver tag enters `.github/workflows/release.yml`, which first proves the ordinary npm package is healthy by running tests, building `dist/`, dry-running `npm pack` against the curated package file list, and then installing the real tarball into a temporary global prefix as a smoke test. That smoke test now deliberately captures only the final line of `npm pack` output, because npm prints prepack build chatter before the filename and the workflow needs the filename alone for the install command. From there the workflow branches into two macOS runners that call `scripts/release/build-sea.mjs`, each one bundling the CLI into a single CommonJS file, generating a Node SEA blob, injecting it into the local `node` executable, signing the result, and packaging the binary as a tarball. Once those artifacts exist, the release workflow gathers them back together, publishes or confirms the npm package, attaches the macOS tarballs and checksum file to the GitHub Release, renders a Homebrew formula through `scripts/release/render-homebrew-formula.mjs`, tests that formula on clean macOS runners, and only then pushes the formula into the dedicated Homebrew tap. In other words, distribution is now treated as another multi-stage interpretation pipeline: source code becomes validated release artifacts, and those artifacts become install surfaces that match the audience they are meant to serve.

## Philosophy

The governing tradeoff in this codebase is semantic clarity over visual fidelity. EPUB is full of markup whose job is to make a reading system render a page attractively. This converter is willing to discard some of that ornamental or layout-oriented structure if doing so yields markdown that is easier to read and easier for downstream tools to reason about. It still tries to preserve important links, headings, lists, tables, and notes, but it treats exact visual reconstruction as secondary.

Another deliberate choice is to separate interpretation from rendering. `src/parse-epub.ts` discovers what the book physically contains, `src/sectioning.ts` decides how that content should be grouped conceptually, and only then do `src/markdown.ts` and `src/output.ts` worry about file formats and bytes on disk. That separation keeps the code honest about which decisions are structural and which are cosmetic. It also makes the repository easier to read in prose, which is a first-class requirement here rather than an afterthought.

The same separation now appears in the shipping story. The regular TypeScript build remains the authoritative npm-facing artifact because it matches the package manager and keeps the CLI transparent to Node users. The standalone binary path is intentionally separate because Node SEA has different constraints: it needs one CommonJS script, not an ordinary ESM module graph on disk. Instead of bending the whole codebase around that constraint, the repository accepts a dual-build philosophy: one output is optimized for package ecosystems that expect JavaScript, and another is optimized for users who simply want a native-feeling executable. That split looks redundant at first glance, but it keeps each distribution channel honest about what it actually needs.
